What Is a Token?
The atomic unit of all LLM computation and cost. Every character, word, and space resolves into tokens — and every token has a price.
Core Concept
LLMs don't process text directly — they process tokens: subword units generated by Byte-Pair Encoding (BPE). A token is roughly 3–4 characters or 0.75 words in English.

  • "Hello world" → 2 tokens
  • "Unbelievable" → 3 tokens (Un / believ / able)
  • Code symbols, whitespace, numbers each tokenise differently
  • Non-English languages use more tokens per word (2–4×)
Why It Matters
  • Every API call is billed per input + output token
  • Context windows have hard token limits (32K–200K+)
  • Token count ≠ word count — estimating by words leads to budget errors
  • Whitespace, punctuation, and formatting all count
  • Tokens drive both cost and latency
Efficient Patterns
  • Dense prose over bullet lists
  • Short variable names in code prompts
  • Remove redundant whitespace
  • Abbreviate known terms
  • Use JSON without extra spacing
Wasteful Patterns
  • Excessive markdown formatting
  • Verbose role descriptions
  • Repeated context injections
  • Raw XML/HTML in prompts
  • Unnecessary examples
Token Ratios by Language
  • English: ~4 chars / token
  • Spanish/French: ~3.5 chars / token
  • Chinese/Japanese: ~1.5 chars / token
  • Code (Python): ~3 chars / token
  • JSON: ~3.5 chars / token

The Output Cost Trap
Output tokens are the largest cost lever most teams ignore. A model asked to "explain in detail" vs "summarise in 3 bullets" can produce 10× more tokens. Always instruct the model on output length and format explicitly.

Token Counting & Measurement
Count tokens before you send them. Pre-flight counting prevents context overflows, enables graceful degradation, and gives you full cost visibility.
Counting Tools
  • tiktoken — OpenAI's fast BPE tokeniser (Python)
  • Anthropic count_tokens API — exact counts pre-send
  • tokenize.js — browser-side counting
  • LangChain token splitters — chunking-aware counting
  • LlamaIndex token counters — pipeline-integrated
What to Log Per Request
  • System prompt tokens (static overhead)
  • History tokens (dynamic, grows over time)
  • RAG context tokens
  • Tool schema tokens
  • User message tokens
  • Output tokens (actual + budgeted)
  • Cache hits vs cache misses

Token Budgets & Limits
Budgets are the governance layer of token optimisation. Define them at every level — request, user, feature, team — and enforce them with graceful degradation.
Soft Budget (Warning)
70–85% of limit. Log a warning, notify the user that context is filling, begin proactive history summarisation. Don't break the experience.
Hard Budget (Enforce)
90% of limit. Truncate oldest history, compress RAG context, remove optional tool schemas. Maintain core functionality at minimum token footprint.
Emergency Limit (Block)
100% of limit. Return a structured error or fallback response. Never send a request that will fail or cost unexpectedly at the API layer.
Budget Dimensions
  • Per-request — max context + max output tokens
  • Per-session — cumulative tokens in a conversation
  • Per-user/day — rate limiting individuals
  • Per-feature — chat vs code gen vs summarisation
  • Per-team/month — departmental chargeback
Graceful Degradation Ladder
  • Step 1: Summarise oldest history turns
  • Step 2: Remove optional tool schemas
  • Step 3: Reduce RAG top-k from 10 → 3
  • Step 4: Strip examples from system prompt
  • Step 5: Route to smaller model
  • Step 6: Return cached/fallback response

Layer 1: System Prompt
Static instructions sent on every request. Optimise once, save forever. Target: <500 tokens. Enable KV caching to reduce cost by 90%.
Layer 2: History
Grows with every turn. Implement sliding window, periodic summarisation, or selective retention to cap at a token budget.
Layer 3: RAG Context
The largest variable layer. Re-ranking, smaller chunks, and budget caps are the primary levers. Target: ≤1,500 tokens of retrieved context.
System Prompt Tokens
System prompts are a silent cost multiplier. Every token in your system prompt is paid on every single request — at scale, even 200 extra words can cost thousands of dollars monthly.
Cost at Scale Example
A 2,000-token system prompt at 100,000 requests/day:

Uncached: 200M tokens × $3/1M = $600/day = $18,000/month
Cached (90% hit): 20M × $3 + 180M × $0.30 = $114/day = $3,420/month

Savings from caching alone: $14,580/month
Compression Checklist
  • Remove "you are a helpful assistant" boilerplate
  • Replace prose rules with bullet directives
  • Eliminate redundant instructions
  • Use abbreviations for repeated terms
  • Move examples to a cacheable prefix block
  • Audit quarterly — rules accumulate over time
Cache Everything Static: Place all static content (persona, rules, examples, documents) at the beginning of the context as a cacheable block. Only dynamic, per-request content should be outside the cache boundary.
Conversation History Management
By default, every new message carries the entire conversation. A 20-turn chat can accumulate 40,000+ tokens — paid again on message 21. This is the #1 hidden cost driver in chat apps.
Strategy 1: Sliding Window
Keep only the last N turns. Simple, predictable, but loses long-term context. Best for task-focused chatbots where recency matters most.
Strategy 2: Summarisation
Periodically compress older turns into a summary paragraph. Preserves key context at ~10% of original token cost. Best for long support conversations.
Strategy 3: Selective Retention
Keep only turns flagged as high-value (user corrections, key decisions, tool results). Score turns by relevance and prune dynamically.
# Summarise history when it exceeds a token budget def manage_history(messages, budget=4000): if count_tokens(messages) < budget: return messages # Summarise all but last 4 turns old = messages[:-4] recent = messages[-4:] summary = summarise(old) # ~200 tokens return [{"role":"user","content":f"[Summary of prior context: {summary}]"}] + recent
Tool Schema Tokens
Every tool definition you include in a request costs tokens — even when that tool is never invoked. A 20-tool agent can waste 3,000+ tokens per request on schema definitions alone.
The Schema Overhead Problem
Typical tool schema sizes:

  • Simple tool (name + desc): ~80 tokens
  • Medium tool (+ params): ~200 tokens
  • Complex tool (+ enum + examples): ~400 tokens
  • 20 complex tools: ~8,000 tokens/request
Lazy Schema Loading
  • Include only tools relevant to the current task
  • Use a router to select the right tool subset
  • Start with name-only schemas; expand on first use
  • Cache schema tokens with KV cache
  • Compress descriptions without losing semantics
Prompt Compression
Systematically reduce prompt length without reducing task quality. Well-compressed prompts often perform better — brevity forces clarity.
LLMLingua-2 Compression
Microsoft's token-budget-aware prompt compressor. Removes semantically redundant tokens while preserving task-critical information. Achieves 4–8× compression with <5% performance drop on most tasks.
Manual Compression Techniques
  • Replace narrative instructions with imperative directives
  • Use structured formats (YAML > verbose JSON > prose)
  • Remove pleasantries and meta-commentary
  • Deduplicate repeated concepts
  • Use domain abbreviations consistently
Before (87 tokens)
"You are a helpful customer service assistant who is always polite and professional. When a customer asks a question, please make sure to answer it thoroughly and provide any relevant information that might help them. Always be friendly and supportive."
After (22 tokens)
"Role: customer support agent. Tone: professional, concise. Answer user questions accurately. Ask for clarification if ambiguous."
Result
75% token reduction. Same (often better) output quality because the model receives cleaner, less ambiguous instructions.
Output Token Control
Output tokens cost 3–5× more than input tokens and are the largest single cost lever most teams haven't pulled. Every token the model generates is money leaving your account.
Always Set max_tokens
Never send a request without max_tokens. The default allows models to generate thousands of tokens for simple tasks. Even setting max_tokens=200 for classification tasks can cut output costs by 90%.
Format-Constrained Output
  • Use JSON mode — eliminates preamble tokens
  • Specify exact output schema in the prompt
  • Request bullet points, not paragraphs
  • Specify maximum length explicitly ("max 3 sentences")
  • Use "Answer only with: yes/no/unsure" for classification
The Verbosity Default: LLMs are trained to be thorough and helpful, which means they default to longer, more detailed responses. Override this explicitly in every prompt. "Be concise" is not enough — specify the exact format and maximum length.
Few-Shot Efficiency
More examples don't linearly improve performance. Find the minimum effective shot count for each task type — most tasks need 1–3 examples when framed correctly.
Example Budget Guidelines
  • Classification tasks: 1–2 examples per class
  • Format transformation: 1 input/output pair
  • Code generation: 0–1 examples (use type hints instead)
  • Complex reasoning: 2–3 chain-of-thought examples
  • Highly specialised domain: 3–5 examples maximum
Cached Example Patterns
Place all few-shot examples in the system prompt (cacheable) rather than repeating them per-request. Cache once, save millions of tokens. Use cache breakpoints to separate static examples from dynamic context.
Structured Output
Constrained generation produces exactly what you need — nothing more. It eliminates preamble, explanation padding, and post-processing extraction calls.
JSON Mode Benefits
  • Eliminates "Sure, here is the JSON you requested:" preamble
  • Removes markdown code fences (```json)
  • No need for a second extraction call
  • Predictable token count for budgeting
  • Directly parseable — zero processing overhead
Token Savings by Format
  • Asking for JSON: 30–50% fewer output tokens
  • Structured schema: 20–40% fewer tokens
  • Bullet list vs prose: 40–60% fewer tokens
  • Yes/No classification: 95% fewer tokens
Prompt Caching (KV Cache)
The single highest-ROI optimisation available. Cache static context at the model layer and pay 90% less on every cache hit. Implement on day one.
How KV Cache Works
The model computes key-value attention matrices for your prompt. With caching enabled, these are stored server-side. Subsequent requests with identical prefix pay only for the cache read (~10% of input cost) instead of full recomputation.
Cache Killers — Avoid These
  • Timestamps or request IDs in system prompt
  • User name/personalisation before static content
  • Dynamic content injected before examples
  • Random ordering of tool schemas
  • Any variation in the static prefix
Architecture Rule: Always place content in order of stability: system instructions → documents → examples → (cache boundary) → conversation history → user message. Cache everything above the boundary.

Semantic Response Caching
Cache full AI responses for semantically similar queries. A vector similarity lookup serves 30–60% of requests with zero LLM cost — the most powerful throughput multiplier available.
How It Works
1. Embed the user query → vector
2. Search Redis/Pinecone for similar cached queries
3. If cosine similarity > threshold (0.92+): return cached response
4. If miss: call LLM, store response + embedding in cache
5. TTL-based expiry for time-sensitive content
When It Excels
  • FAQ and support bots (high query repetition)
  • Document Q&A on stable content
  • Product description generation
  • Code explanation for common patterns
  • High-traffic consumer applications
SIMILARITY_THRESHOLD = 0.92 async def cached_completion(query: str) -> str: query_vec = await embed(query) hit = await redis.vector_search(query_vec, threshold=SIMILARITY_THRESHOLD) if hit: metrics.increment("cache_hits") return hit.response response = await llm_call(query) await redis.store(query_vec, response, ttl=3600) return response
Tool Result Memoization
Agent loops often call the same tools with identical parameters multiple times. Memoising results eliminates redundant API calls and dramatically reduces context pollution from repeated tool outputs.
Within-Session Cache
Hash (tool_name + params) → cache result for the duration of an agent run. A weather lookup, database query, or API call performed in iteration 2 need not re-execute in iteration 5.
Cross-Session Cache
For stable data (product catalogues, configuration, user profiles), persist tool results in Redis with appropriate TTL. Saves both token costs and downstream API costs simultaneously.
Embedding Caching
Embedding costs are often overlooked — but re-embedding unchanged documents at query time is pure waste. Persist vectors; only recompute on actual content changes.
Storage Strategy
  • Store document hash → embedding vector
  • On update: diff hashes, re-embed only changed chunks
  • Use pgvector, Pinecone, or Weaviate for persistent storage
  • Cold-start: bulk embed entire corpus once
  • Warm path: hash check before any embed call
Cost Savings
A 10,000-document corpus embedded daily costs ~$1/day with text-embedding-3-small. With caching and differential updates (5% daily change rate), cost drops to ~$0.05/day — 95% reduction.
Chunk Sizing Strategy
Chunk size directly controls RAG context quality and cost. Too large: bloated context with low signal. Too small: fragmented context loses coherence. The optimal size is task-dependent.
Too Large (>1000 tokens)
High token cost, low retrieval precision, relevant info buried in noise. Retrieval similarity becomes diluted across multiple topics per chunk.
Sweet Spot (200–400 tokens)
Balanced precision and coherence. Works for most document types. Combine with sentence-aware splitting to avoid mid-sentence breaks.
Too Small (<50 tokens)
High retrieval noise, fragments miss context, requires larger top-k to find relevant content. Often worse retrieval quality than larger chunks.
Hybrid Chunking
  • Split on semantic boundaries (paragraphs, sections)
  • Add 10–20% overlap to prevent context cliffs
  • Store parent chunk → retrieve child chunks
  • Proposition chunking: one factual claim per chunk
  • Code: split by function/class, not arbitrary length
Metadata Injection
Prepend source, section title, and page number to each chunk (<30 tokens). This improves retrieval precision and allows context attribution without embedding these in the chunk content itself.
Re-ranking & Precision Retrieval
Retrieve broadly with vector search, then filter aggressively with a re-ranker. Send top-3 high-precision chunks instead of top-20 noisy ones. Same quality, 80% fewer context tokens.
Two-Stage Pipeline
Stage 1 — Vector Recall: Retrieve top-50 candidates by embedding similarity (fast, approximate).

Stage 2 — Cross-Encoder Re-rank: Score each candidate against the query with a cross-encoder model (slower, precise). Keep top-3 to top-5.

Result: 10× better precision at 20% of the context cost.
Re-ranker Options
  • Cohere Rerank — hosted, easiest integration
  • BGE-Reranker — open source, self-hostable
  • cross-encoder/ms-marco — HuggingFace, free
  • FlashRank — ultra-fast, minimal dependencies
  • LLM-as-reranker — highest quality, higher cost
RAG Context Budget Control
Set a hard token budget for all retrieved context and enforce it before sending. RAG context is the most controllable cost layer — it should never exceed its allocation.
Budget Allocation
For an 8K token request budget:

System prompt: 500 tokens
History: 1,500 tokens
RAG budget: 1,500 tokens
Tool schemas: 500 tokens
User message: 500 tokens
Output reserved: 3,500 tokens

RAG content beyond 1,500 tokens is truncated by relevance score.
Enforcement Strategies
  • Count retrieved chunks before injection
  • Sort by relevance score, take until budget exhausted
  • Truncate individual chunks at token boundary
  • Prefer fewer complete chunks over many truncated ones
  • Log budget utilisation per request
Summary Indexing & RAPTOR
Build a hierarchy of summaries from your corpus. Retrieve a compact 300-token summary instead of 3,000 tokens of raw chunks for high-level queries.
RAPTOR Architecture
Level 0: Original chunks (200–400 tokens)
Level 1: Cluster summaries (100–200 tokens, ~10 chunks per cluster)
Level 2: Section summaries (50–100 tokens)
Level 3: Document summaries (30–50 tokens)

Query routing decides which level to retrieve from based on query specificity.
Query Routing Logic
  • General question → top-level summary (50 tokens)
  • Section-specific → level 1 summary (150 tokens)
  • Detail-specific → leaf chunks (300 tokens)
  • Mixed → hybrid retrieval across levels

Control Mechanisms
  • Hard iteration limit (never exceed 10 iterations)
  • Token budget per loop (abort if exceeded)

class BudgetedAgent: def __init__(self, max_iter=8, token_budget=40000): self.max_iter = max_iter; self.token_budget = token_budget self.total_tokens = 0; self.iter = 0 def step(self, messages): if self.iter >= self.max_iter: raise AgentLimitError("Max iterations reached") estimated = count_tokens(messages) if self.total_tokens + estimated > self.token_budget: messages = self.compress_history(messages) result = llm_call(messages) self.total_tokens += result.usage.total_tokens self.iter += 1 return result
Multi-Agent Overhead
Every agent-to-agent message serialises full context. An orchestrator managing 5 sub-agents can easily generate 5× the token cost of a single-agent approach if not architected carefully.
Where Cost Explodes
  • Orchestrator passes full context to each sub-agent
  • Sub-agents return full conversation history
  • Context duplicated across all parallel branches
  • Each agent re-loads tool schemas independently
  • No shared memory — same research done repeatedly
Efficient Architecture
  • Pass only task-relevant context to sub-agents
  • Sub-agents return structured summaries, not full history
  • Use a shared read-only memory store
  • Centralised tool schema registry (cached)
  • Orchestrator compresses sub-agent outputs before aggregation
Planning & Reasoning Costs
Extended thinking, chain-of-thought, and planning steps generate thousands of tokens before the actual response begins. Budget reasoning tokens as a first-class cost category.
Reasoning Token Reality
Claude's extended thinking can generate 5,000–20,000 thinking tokens per request. At Opus 4 output pricing ($75/1M), 20,000 thinking tokens = $1.50 per request — before the actual response.
Budget_tokens Parameter
Anthropic's thinking API accepts a budget_tokens parameter to cap reasoning length. Set this based on task complexity: simple tasks need 1,000 tokens of thinking; complex reasoning rarely benefits beyond 10,000.
When to Use Extended Thinking
  • Multi-step mathematical reasoning
  • Complex code architecture decisions
  • Long-horizon planning tasks
  • Tasks requiring self-verification
When NOT to Use It
  • Simple extraction or classification
  • Formatting or transformation tasks
  • Factual retrieval from provided context
  • Any task a standard call handles correctly
Tool Call Result Management
Raw API results injected directly into context are the most common source of token waste in agentic systems. A 50KB API response contains ~12,000 tokens — extract the 200 tokens you actually need.
Extraction Pattern
Define a result schema for each tool call. After receiving the result, run a lightweight extraction step that pulls only the fields needed for the current task before appending to agent history. Never append raw API responses.
Compact Serialisation
  • Remove null/empty fields from JSON results
  • Use short field names in tool result schemas
  • Truncate long string fields at character limit
  • Aggregate arrays: "47 items, first 3: [...]"
  • Return status codes, not full error messages
Intelligent Model Routing
Not every task needs a frontier model. Route 80% of requests to fast, cheap models — reserve Opus-tier only for tasks that genuinely require it. Potential 80% cost reduction on model spend.
Routing Tiers
Tier 1 (Micro): Classification, extraction, simple Q&A → Haiku 3.5 or GPT-4o mini

Tier 2 (Standard): Generation, summarisation, code completion → Sonnet 4 or GPT-4o

Tier 3 (Advanced): Complex reasoning, novel problems, high-stakes → Opus 4 or o1
Routing Signals
  • Task type (classify, generate, reason)
  • Estimated output length
  • Domain complexity score
  • User tier / SLA requirement
  • Confidence from a cheap classifier model

Speculative Decoding
A small draft model generates token candidates in parallel; the large model verifies and corrects them. Same output quality at 2–4× higher throughput and lower effective latency.
How It Works
1. Small draft model (e.g., Haiku) proposes K tokens
2. Large model evaluates all K tokens in parallel
3. Accepted tokens are kept; first rejection restarts
4. Net effect: 2–4× speed improvement on simple continuations
5. Best for code completion, template filling, factual continuation
Cost-Latency Trade-off
Speculative decoding doesn't reduce total tokens — it reduces wall-clock latency. The cost is similar to running both models. Value is in user experience and throughput capacity, not direct token savings.
Fine-Tuning for Prompt Reduction
Bake repetitive instructions, style, and domain knowledge into the model weights. Eliminate 80% of system prompt tokens for high-volume, predictable tasks. Best ROI at 50,000+ requests/month.
What Fine-Tuning Eliminates
  • Role and persona descriptions
  • Output format instructions
  • Domain-specific terminology definitions
  • Company-specific rules and constraints
  • Few-shot examples baked into weights
Break-Even Analysis
Fine-tuning cost: ~$500–$2,000 one-time.

Savings: 1,500-token system prompt × 100K requests/month × $0.003 = $450/month.

Break-even: <4 months. ROI thereafter: infinite.

  • Embedding generation for new content

Rate Limiting & Quotas
Token quotas are the first line of defence against runaway costs and abuse. Define limits at every level and enforce them before the API call — not after the bill arrives.
Quota Hierarchy
Per-request: Max input + output tokens hard limit
Per-user/hour: Prevent individual abuse
Per-user/day: Fair use enforcement
Per-feature/day: Budget allocation per product area
Per-team/month: Departmental budget cap
Platform total: Global circuit breaker
Graceful Quota Responses
  • Warn at 80% of daily quota
  • Throttle to reduced limits at 90%
  • Return cached/static responses at 100%
  • Show quota status in user-facing UI
  • Allow quota increases via self-service portal
Cost Attribution & Chargeback
Attribute every token to a cost centre. Showback drives awareness; chargeback drives accountability. Both are necessary to make AI costs manageable at organisational scale.
Tagging Strategy
Tag every API request with:

team, product, feature, environment, user_tier, experiment_id

Route tags to your observability platform (Datadog, Grafana, AWS Cost Explorer). Generate monthly cost-by-dimension reports automatically.
Chargeback Models
  • Showback: Report costs to teams, no billing
  • Soft chargeback: Internal credits, no real money
  • Hard chargeback: Real budget impact on team P&L
  • Start with showback — accountability emerges naturally
Token Anomaly Detection
Cost spikes happen fast. An agent loop bug, a prompt injection, or a traffic surge can generate thousands of dollars in unexpected charges in minutes. Automated anomaly detection is non-optional at scale.
Anomaly Signals to Monitor
  • Request token count > 3× rolling average
  • Single user consuming >10% of daily budget
  • Agent iteration count exceeds threshold
  • Cost/hour spike >2× standard deviation
  • Cache hit rate drops >20% unexpectedly
Response Actions
  • Alert: Slack/PagerDuty notification
  • Throttle: Reduce quota for offending entity
  • Block: Stop requests from suspected abuse
  • Investigate: Flag request for human review
  • Rollback: Revert recent prompt/model changes

0 / 10 complete
Audit your system prompt. Count the tokens. Remove every sentence that doesn't change model behaviour. Target: under 500 tokens.
P1
Enable prompt caching. Add cache_control to your system prompt and any static documents. Immediate 60–90% cost reduction on those tokens.
P1
Set max_tokens on every API call. Never rely on model defaults. Set a realistic limit for each use case — classification tasks should be 50–200 tokens, not 4,096.
P1
Add token logging to every LLM call. Log input_tokens, output_tokens, cache_hit, and cost_usd with a feature tag. You cannot optimise what you don't measure.
P1
Remove boilerplate from prompts. Delete "You are a helpful assistant", "Please", "Thank you", and similar non-functional filler. Usually saves 50–150 tokens per request.
P1
Use JSON mode for structured outputs. Eliminates preamble tokens and markdown fences. Reduces output tokens 30–50% for data extraction tasks.
P2
Implement conversation history truncation. Cap history at a token budget (e.g., 2,000 tokens). Sliding window or summarisation — either is better than unlimited growth.
P2
Reduce RAG top-k from the default. Most defaults are top-10 or top-20. Reduce to top-5 with a relevance threshold filter. Cuts RAG context tokens 50–75%.
P2
Identify your top 3 most expensive request patterns. Sort by total_tokens descending in your logs. The top 3 often account for 50%+ of spend.
P2
Add pre-flight token counting. Count tokens before sending — abort or truncate if over budget. Prevents unexpected context overflow costs.

🔧 Engineering Implementation — 1–2 Weeks
0 / 10 complete
Deploy semantic response caching. Redis + embedding similarity lookup. Start with a 0.92 cosine threshold. Target: 30%+ cache hit rate within 2 weeks.
P1
Add a re-ranker to your RAG pipeline. Retrieve top-50, re-rank, send top-5. Cohere Rerank or BGE-Reranker are easy starting points. Reduces RAG context 60–80%.
P1
Implement model routing. Build a simple classifier that routes requests to Haiku (simple), Sonnet (medium), or Opus (complex). Start with rule-based routing before ML-based.
P1
Add per-user token quotas. Implement daily token budgets per user tier. Warn at 80%, throttle at 90%, block at 100% with a graceful message.
P2
Implement tool result memoization. Cache tool call results within agent sessions by (tool_name + params) hash. Especially impactful for search, lookup, and API calls.
P2
Add agent loop limits. Hard cap at 8–10 iterations per task. Implement a token budget that triggers mid-loop compression if exceeded.
P2
Implement lazy tool schema loading. Don't include all tools in every request. Use a lightweight router to select the 3–5 most relevant tools per task type.
P2
Set up cost anomaly alerts. Alert when any request exceeds 3× the rolling average token count, or when hourly cost spikes above 2 standard deviations.
P2
Move eligible workloads to batch API. Identify classification, summarisation, and enrichment pipelines that don't require real-time responses. 50% cost reduction immediately.
P3
Implement conversation summarisation. For long sessions, periodically summarise old turns into a compact paragraph. Compress 2,000 tokens of history to 200 tokens without losing key context.
